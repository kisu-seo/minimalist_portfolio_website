# [핵심 원칙]
- 설명은 이해하기 쉽게 3줄 이하 불릿 형태로 핵심만 설명해 줘.
- **섹션 주석 명시:** 코드 작성 시 해당 코드가 속한 섹션/영역을 주석으로 반드시 표시해줘.
  - 형식: `/* === [섹션명] === */` 또는 `// --- [영역명] ---`
  - 예시: `/* === Header / Navigation === */`, `// --- Form Validation ---`
- **접근성(A11y) First:** 모든 HTML은 스크린 리더가 읽을 수 있도록 접근성을 고려해서 짜줘 (aria-label, 시멘틱 태그 활용).
- **언어 일관성:** 기술적인 키워드를 제외한 모든 가이드와 대화는 한국어로 진행하며, 영어로 답변이 나올 경우 즉시 한국어로 재번역해서 알려줘.

# 핀셋 수정 강제 지침 (Strict Modification Rules)
코드 수정 시 아래의 원칙을 **예외 없이** 엄격하게 준수하십시오.
* **수정 타겟 고정:** 내가 명시적으로 요청한 [수정 대상 기능/오류 부분]의 코드만 정확히 수정하세요.
* **주변 코드 보존:** 요청 내용과 직접적인 관련이 없는 다른 컴포넌트, 스타일(Tailwind 클래스), 구조, 변수명은 **단 한 글자도 변경하지 마세요.**
* **자율적 리팩토링 금지:** 코드를 더 '깔끔하게' 만들거나 '최적화'하려는 자의적인 판단과 숨은 수정을 절대 금지합니다.

# [하네스 워크플로우 지침 (Harness Execution)]
에러를 최소화하고 안정적인 빌드를 위해 다음 자동화 및 관리 루프를 강제합니다.

## 1. 하네스 파일 관리
프로젝트 루트에서 아래 파일들을 관리하며 작업합니다. 파일이 존재하지 않으면 임의로 생성하지 말고 먼저 사용자에게 새로 만들지 확인하십시오.

- **`PROGRESS.md` (진행 상황):** 작업 시작 전 먼저 읽어 컨텍스트를 파악하고, 완료된 태스크는 체크(`[x]`)하십시오.
- **`MEMORY.md` (에러/교훈 로그):** 작업 시작 전 먼저 읽고, 디버깅 과정에서 얻은 교훈은 종료 후 한 줄로 즉시 기록하십시오.
  - 기록 형식 예시: `- [날짜] OOO 챌린지: flex 정렬 시 align-items 누락으로 세로 정렬 깨짐`
- **`STACK.md` (기술 스택/의존성 고정):** 프레임워크, 주요 라이브러리, 버전을 기록합니다. 사용자 승인 없이 새 라이브러리를 추가하거나 버전을 임의로 변경하지 마십시오. React 등 프레임워크 프로젝트를 시작할 때 가장 먼저 이 파일을 확인/작성하십시오.
- **`DECISIONS.md` (구조적 결정 기록):** 폴더 구조, 상태 관리 방식, 컴포넌트 분리 기준 등 프로젝트의 구조적 결정을 기록합니다. 이미 기록된 결정과 다른 방식이 필요하다고 판단되면 임의로 바꾸지 말고 먼저 사용자에게 알리고 확인받으십시오.
- **`GLOSSARY.md` (용어 사전):** 비전공자 학습자를 위해, 새로 등장한 기술 용어(예: flex, z-index, 상태 관리 등)를 짧은 한국어 설명과 함께 기록합니다. 같은 용어가 이미 기록되어 있다면 반복 설명 대신 이 파일을 참고하도록 안내하십시오.
- **`HANDOFF.md` (도구 간 인계 메모):** 클로드 코드와 안티그래비티 등 서로 다른 AI 도구를 오갈 때 사용합니다. 작업 종료 시 "지금까지 한 일 / 다음에 할 일 / 주의할 점"을 3줄 이내로 남기고, 작업 시작 시 이 파일을 먼저 읽어 이어서 진행하십시오.

## 2. 디자인 시안 & 스타일 가이드 동기화
- `style-guide.md` 파일이 프로젝트에 있으면, 그 안의 CSS 변수(컬러, 폰트 세팅)를 최우선으로 조회하여 적용하고 임의의 헥사코드(Hex code) 사용을 금지합니다.
- `style-guide.md`가 없는 프로젝트라면, 작업 시작 전 사용자에게 디자인 시안 도구(피그마 등)에서 디자인 토큰(색상/폰트/여백)을 추출해 전달해달라고 먼저 요청하십시오. 토큰이 불명확한 채로 임의 추측하지 마십시오.
- 반응형 브레이크포인트는 프로젝트의 `README.md`에 명시된 값을 최우선으로 따르십시오.
  - README에 별도 명시가 없을 경우에만 기본값(모바일 375px → 태블릿 768px → 데스크톱 1024px)을 사용하십시오.
  - 항상 모바일 레이아웃을 완전히 끝낸 후, 미디어 쿼리로 태블릿 → 데스크톱 순으로 확장하십시오.

## 3. 자동화된 검증 루프
- 코드 수정/생성 완료 후, 사용자에게 보고하기 전에 아래 순서로 스스로 검증하십시오.
  1. **빌드 스크립트 존재 확인:** `package.json`에 `lint`/`build` 스크립트가 있는 경우에만 `npm run lint`, `npm run build`를 실행하십시오. 순수 HTML/CSS/JS 프로젝트처럼 해당 스크립트가 없는 경우, 이 단계를 건너뛰고 문법 오류 여부만 육안으로 재확인하십시오.
  2. **에러 발생 시:** 사용자 개입 없이 에러 로그를 분석해 스스로 수정하는 루프를 **최대 3회**까지 반복하십시오.
  3. **3회 후에도 실패 시:** 더 이상 자율 반복하지 말고 즉시 작업을 멈추고, 시도한 내용과 마지막 에러 메시지를 사용자에게 3줄 이내로 요약 보고하십시오.

## 4. 완료 기준 검증 (Design QA)
- "완료"를 보고하기 전, 아래 항목을 스스로 점검하고 결과를 3줄 이내 불릿으로 보고하십시오.
  - (시안이 있는 경우) 시안 대비 레이아웃/색상/폰트/간격 일치 여부
  - (시안이 없는 경우) 여백·정렬·폰트 크기 등이 페이지 내에서 일관되게 적용되었는지, 요청한 요구사항을 빠짐없이 반영했는지 여부
  - 모바일/태블릿/데스크톱 반응형 동작 여부
  - 접근성 기본 항목(대체텍스트, 시멘틱 태그, 키보드 포커스) 여부
- 시각적으로 비교 가능한 스크린샷이 있다면, 시안 이미지와 나란히 비교해 차이점을 먼저 짚어준 뒤 보고하십시오.

# [피드백 및 디버깅 스타일]
- 에러가 나면 "문법이 틀렸어요"보다는 **"구조(Structure)와 흐름(Flow)"** 관점에서 핵심만 불릿 형태로 3줄 이내로 설명해 줘.
